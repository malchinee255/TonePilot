# TonePilot 管理端多来源知识采集设计

## 背景

TonePilot 的知识库不应该只依赖手工录入或样片分析。真实的调色知识来源很多，包括抖音调色教程、摄影师或大师的 Lightroom 调色记录、XMP/预设文件、人工整理的规则和已有样片分析结果。

因此管理端需要抽象出“知识来源”能力：先管理来源和原始素材，再通过 Agent 抽取成结构化调色知识，最后进入现有知识审核池。抖音导入只是其中一个来源适配器。

## 目标

1. 在管理端新增统一的知识来源管理能力，支持多种来源。
2. 将外部素材先沉淀为可追溯的原始材料，再转换为可审核知识。
3. 保留现有知识审核流程，避免直接把不可靠内容发布给运行时 Agent。
4. 为未来扩展大师调色记录、XMP、预设、合集批量采集预留边界。
5. 本地运行时只消费已审核知识，不承担采集和清洗职责。

## 非目标

1. 第一阶段不实现抖音账号全量爬取，只支持管理员录入视频链接或批量链接。
2. 第一阶段不绕过平台权限，不保存完整视频作为产品资产，默认保存字幕、摘要、来源信息和结构化知识。
3. 第一阶段不把知识向量检索做成独立服务，先复用现有管理端知识库和 RAG 检索能力。
4. 第一阶段不支持 Lightroom 局部蒙版知识的自动提取，只处理全局调色经验和参数变化。

## 功能设计

### 知识来源

管理端新增“知识来源”页面，用来创建和查看不同来源的采集任务。

支持来源类型：

- `douyin_video`：抖音调色教程视频。
- `master_edit_record`：大师或摄影师的调色记录、XMP、预设、前后参数差异。
- `manual_note`：管理员手工整理的经验。
- `style_sample`：已有样片分析结果。

知识来源需要记录：

- 来源类型
- 来源标题
- 作者或账号
- 原始链接或文件标识
- 关联风格
- 来源状态
- 采集时间
- 最近处理结果

### 原始素材

每个知识来源可以产生一个或多个原始素材。素材不直接等于知识。

素材类型：

- `transcript`：视频字幕文本。
- `summary`：视频摘要。
- `lightroom_params`：Lightroom 参数快照。
- `param_delta`：调色前后参数差异。
- `xmp`：XMP 或预设内容。
- `manual_text`：手工经验文本。

素材需要保留来源信息、处理版本和原始内容摘要，方便后续追溯。

### 抽取任务

管理端新增知识抽取任务，用于把素材转换成待审核知识。

任务状态：

- `pending`：等待处理。
- `running`：处理中。
- `succeeded`：抽取成功。
- `failed`：抽取失败。
- `review_required`：已生成待审核知识。

任务记录：

- 来源 ID
- 素材 ID
- 使用的模型供应商
- Prompt 版本
- Agent 输出
- 错误信息
- 生成的知识 ID

### Agent 抽取输出

所有来源最终转换成统一的结构化知识草稿。

建议字段：

- 标题
- 适用场景
- 目标风格
- 常见问题
- 调色策略
- Lightroom 参数建议
- 禁忌规则
- 适用条件
- 置信度
- 来源摘要

示例：

```json
{
  "title": "夜景电影感压高光策略",
  "scene": "夜景 / 城市灯光 / 暗光",
  "targetStyle": "夜景电影感",
  "problems": ["灯牌过曝", "暗部死黑", "绿色偏脏"],
  "strategy": [
    "优先压低高光，保留灯牌和天空层次",
    "适度提高阴影，恢复暗部可读性",
    "降低绿色饱和度，避免城市绿植发脏",
    "用户没有明确要求时，不主动修改白平衡"
  ],
  "lightroomHints": {
    "basic": ["highlights", "shadows", "contrast", "dehaze"],
    "hsl": ["greenSaturation", "blueSaturation", "orangeLuminance"]
  },
  "confidence": 0.82
}
```

## 数据流

```text
管理员创建知识来源
  -> 采集适配器获取原始素材
  -> 存储素材和来源元数据
  -> Agent 抽取结构化调色知识
  -> 生成待审核知识
  -> 管理员审核通过
  -> 发布到知识库
  -> 本地运行时同步或检索已审核知识
```

## 抖音导入适配器

抖音导入第一阶段采用外部工具 `video-to-subtitle-summary-skill`，不直接把抓取逻辑内嵌到管理端后端。

最小流程：

1. 管理员输入一个或多个抖音视频链接。
2. 管理端创建 `douyin_video` 类型知识来源。
3. 后端调用采集脚本或异步任务：
   - 解析视频下载直链。
   - 下载视频或音频。
   - 使用 `faster-whisper` 或火山 ASR 生成字幕。
   - 生成字幕文本素材。
4. 管理端 Agent 将字幕转成调色知识草稿。
5. 草稿进入现有知识审核池。

配置项：

- `knowledge.ingestion.douyin.enabled`
- `knowledge.ingestion.douyin.skillPath`
- `knowledge.ingestion.douyin.provider`
- `knowledge.ingestion.asr.backend`
- `knowledge.ingestion.workspace`

失败处理：

- API Key 缺失时，任务失败并提示管理员配置。
- 视频解析失败时，保留来源记录和失败原因。
- ASR 失败时，可重新运行任务。

## 大师调色记录适配器

大师调色记录第一阶段支持管理员上传文件，不自动读取用户本地 Lightroom。

输入内容：

- XMP 文件
- Lightroom 预设
- 调色前后参数 JSON
- 参数差异 CSV
- 手工说明文本

抽取重点：

- 参数变化幅度
- 适用照片场景
- 风格目标
- 哪些参数是核心参数
- 哪些参数属于条件性修改
- 哪些参数不能默认修改

输出仍然进入统一知识审核池。

## 后端边界

管理端后端负责：

- 知识来源、素材、抽取任务的持久化。
- 调用采集适配器。
- 调用大模型抽取结构化知识。
- 审核、发布、禁用知识。
- 向本地运行时提供已审核知识。

本地运行时负责：

- Lightroom 当前照片读取。
- 用户对话和本地 Agent 编排。
- 使用已审核知识辅助调色。
- 调用 Lightroom 插件应用参数。

本地运行时不负责：

- 抖音视频采集。
- 大规模素材清洗。
- 管理员审核。
- 云端知识库维护。

## 前端页面

管理端建议拆成五个视图：

1. `风格管理`：维护风格定义。
2. `知识来源`：新增、查看、重跑来源采集和抽取任务。
3. `知识审核`：审核、编辑、发布或禁用知识。
4. `样片与调色记录`：上传样片、XMP、预设和参数差异。
5. `评测与观测`：查看 Agent 调用、知识命中和评测结果。

第一阶段可以在现有知识库页面中增加“来源导入”区域，不必立刻重做完整导航。

## 持久化建议

新增表：

- `knowledge_source`
- `knowledge_material`
- `knowledge_extraction_job`

复用表：

- `style_knowledge`
- `color_style`
- `style_sample`

`style_knowledge` 需要保留来源引用：

- `source_id`
- `material_id`
- `extraction_job_id`

如果第一阶段不改动现有表结构，可以先把来源引用写入 `metadata` 字段，第二阶段再迁移成显式字段。

## 可观测性

每次抽取任务需要记录：

- 来源类型
- 输入素材摘要
- 使用模型
- Prompt 版本
- 生成知识数量
- 成功或失败原因
- 处理耗时

这些数据可以进入现有观测体系，用于后续评估不同来源知识对调色结果的贡献。

## 第一阶段实施范围

第一阶段实现一个最小可用闭环：

1. 后端新增知识来源、素材、抽取任务模型。
2. 后端新增知识来源 API。
3. 管理端前端新增“抖音视频链接导入”表单和来源列表。
4. 接入 `video-to-subtitle-summary-skill` 的字幕输出能力。
5. 将字幕文本抽取成 `StyleKnowledge` 待审核记录。
6. 保持现有审核流程不变。

## 验收标准

1. 管理员可以创建抖音视频来源。
2. 系统可以生成字幕素材或显示明确失败原因。
3. 系统可以从字幕生成待审核调色知识。
4. 管理员可以审核通过知识。
5. 本地运行时只检索已审核知识。
6. 每条知识能追溯到来源视频或调色记录。
