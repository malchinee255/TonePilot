# TonePilot ReAct 与混合检索运行时设计

## 目标

用户端 Lightroom Local Runtime 需要从“一次模型调用后直接修图”升级为 ReAct 风格的自主 Agent。管理端负责知识库、素材导入、可观测数据和混合检索能力，本地运行时负责读取 Lightroom 状态、组织上下文、调用模型推理、按模型决策调用工具，并把每轮输入、模型输出、工具结果和效果记录回管理端。

## 架构边界

- Lightroom 插件：只负责写入当前选中照片、预览图、当前 Develop Settings，并执行运行时下发的 Lightroom 任务。
- Local Runtime：用户端核心 Agent。它维护当前照片会话、ReAct 步骤、模型配置、Lightroom 工具调用和本地短期缓存。
- TonePilot Admin：远程管理端。它管理知识素材、调色知识、混合检索、向量模型配置、向量库配置、会话观测和运行日志。
- 数据库与向量库：管理端持久化 MySQL，向量检索优先面向 Milvus，同时保留内存向量索引用于开发环境。

## ReAct 行为

运行时的每轮用户输入会产生一组可见的行动轨迹，而不是固定工作流：

1. 主 Agent 读取当前 Lightroom 照片和上一轮调色上下文。
2. 主 Agent 自主决定是否需要检索知识库。
3. 主 Agent 基于照片状态、用户意图、知识命中和历史版本生成调色判断。
4. 主 Agent 决定继续追问用户、只给建议、生成参数，或调用 Lightroom 全局调色工具。
5. 如果调用工具，运行时记录工具结果，并把最终观察写入本轮响应。

UI 只展示“主 Agent 的思考摘要”和“行动轨迹”，不展示写死的流程清单。

## Hybrid RAG

管理端检索服务同时支持：

- 关键词检索：用于命中明确风格、参数名称、教程关键词。
- 向量检索：用于语义相近的调色经验召回。
- 元数据过滤：按照片类型、风格、来源、参数族、素材类型过滤。
- Rerank：对关键词和向量召回结果统一重排。

第一版实现可插拔接口和开发态内存向量索引；Milvus、真实 embedding provider 和 rerank provider 通过配置接入，不阻塞本地联调。

## 模型和向量配置

运行时聊天模型支持 OpenAI 与 Qwen2。管理端知识库向量模型单独配置，不和聊天模型强绑定：

- OpenAI embedding：默认 `text-embedding-3-small`。
- Qwen/DashScope embedding：配置项保留，默认不写死强依赖。
- 本地 embedding：作为离线部署扩展点。

## 错误处理

- 没有配置聊天模型时，运行时返回明确错误，不再回退本地规则。
- 知识库不可用时，Agent 继续基于 Lightroom 状态和用户上下文推理，但响应中写明“知识库未启用或检索失败”。
- Lightroom 工具调用失败时，保留模型方案和参数，不吞掉错误，方便用户重试。
